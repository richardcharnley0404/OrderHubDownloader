# DPOF File Format Reference

DPOF (Digital Print Order Format) is the protocol used to communicate print job instructions to photo print controllers. OrderHub Downloader generates DPOF version 1.00 files.

The generated file is always named **`DPOF.001`** and placed at the root of the order folder.

---

## File Structure

A DPOF file consists of:

1. One `[HDR]` (header) section — contains order-level metadata and controller settings
2. One `[JOB]` section per image/line item — contains per-print instructions

```
[HDR]
... header fields ...

[JOB]
... job fields for image 1 ...

[JOB]
... job fields for image 2 ...
```

---

## [HDR] Section — Field Reference

The header is generated by `DPOFGenerator.generateHeader()`.

```
[HDR]
GEN REV = 01.00
GEN CRT = "OHD" 1.00
GEN DTM = 2026:02:21:15:30:45
USR NAM = "Richard Charnley"
USR CID = "100456"
AUTO CORRECT = 1
VUQ RGN = BGN
VUQ VNM = "NORITSU KOKI" -ATR "QSS-3901"
VUQ VER = 01.00
PRT PSL = B -PSIZE "6x4"
PRT PCH = 145
VUQ RGN = END
```

| Field | Example | Source | Description |
|-------|---------|--------|-------------|
| `GEN REV` | `01.00` | Hardcoded | DPOF standard revision number |
| `GEN CRT` | `"OHD" 1.00` | Hardcoded | Creator identifier — "OHD" = OrderHub Downloader |
| `GEN DTM` | `2026:02:21:15:30:45` | `new Date()` | Generation timestamp in `YYYY:MM:DD:HH:MM:SS` format |
| `USR NAM` | `"Richard Charnley"` | `job.customerName` | Customer name from the OrderHub job |
| `USR CID` | `"100456"` | `job.orderNumber` | Order number — used as the user/customer ID |
| `AUTO CORRECT` | `1` | `controller.autoCorrect` | `1` = enable auto colour correction; `0` = disable |
| `VUQ RGN` | `BGN` / `END` | Hardcoded | Vendor Unique Qualifier region delimiters |
| `VUQ VNM` | `"NORITSU KOKI" -ATR "QSS-3901"` | `controller.vendorName` + `controller.vendorAttribute` | Vendor name and model attribute |
| `VUQ VER` | `01.00` | Hardcoded | Vendor qualifier version |
| `PRT PSL` | `B -PSIZE "6x4"` | `channel.printStyleCode` + first `lineItem.width`x`lineItem.height` | Print style code and paper size (from first line item) |
| `PRT PCH` | `145` | `channel.channelNumber` | Physical paper channel number on the controller |

### AUTO CORRECT values

| Value | Meaning |
|-------|---------|
| `1` | Controller applies automatic colour/exposure correction |
| `0` | No correction — prints as-is |

### PRT PSL — Print Style Code

The print style code (`B`) is controller-specific. Common values:

| Code | Typical meaning |
|------|----------------|
| `B` | Standard borderless print (Noritsu) |
| `S` | Standard with border |

Check your controller documentation for supported codes.

## [JOB] Section — Field Reference

One `[JOB]` section is generated per line item by `DPOFGenerator.generateJob()`.

```
[JOB]
PRT PID = 001
PRT TYP = STD
PRT QTY = 2
IMG FMT = EXIF2 -J
<IMG SRC = "../IMAGES/photo-001.jpg">
VUQ RGN = BGN
VUQ VNM = "NORITSU KOKI" -ATR "QSS-3901"
VUQ VER = 01.00
PRT CVP1 = 1 -STR ""
PRT CVP2 = 1 -STR "ORD-2026-0456"
IMG ORG = "o100456_8x12GLOSS/IMAGES/photo-001.jpg"
VUQ RGN = END
```

| Field | Example | Source | Description |
|-------|---------|--------|-------------|
| `PRT PID` | `001` | `lineItem.lineItemNumber` (zero-padded to 3 digits) | Print job ID — sequential within the DPOF file |
| `PRT TYP` | `STD` | Hardcoded | Print type — `STD` = standard print |
| `PRT QTY` | `2` | `lineItem.quantity` | Number of copies to print |
| `IMG FMT` | `EXIF2 -J` | Hardcoded | Image format — `EXIF2` with JPEG (`-J`) flag |
| `<IMG SRC>` | `"../IMAGES/photo-001.jpg"` | `lineItem.filename` | Relative path from DPOF.001 to the image file |
| `VUQ RGN` | `BGN` / `END` | Hardcoded | Vendor qualifier region delimiters (repeated per job) |
| `VUQ VNM` | `"NORITSU KOKI" -ATR "QSS-3901"` | `controller.vendorName/Attribute` | Repeated vendor metadata |
| `VUQ VER` | `01.00` | Hardcoded | Vendor qualifier version (repeated) |
| `PRT CVP1` | `1 -STR ""` | Hardcoded | Custom variable print 1 — empty string |
| `PRT CVP2` | `1 -STR "ORD-2026-0456"` | `job.orderReference` | Custom variable print 2 — order reference (printed on back or border) |
| `IMG ORG` | `"o100456_8x12GLOSS/IMAGES/photo-001.jpg"` | `job.orderNumber` + `job.productCode` + `lineItem.filename` | Original image path reference — includes the order folder name |

### IMG SRC path

The `<IMG SRC>` tag uses a relative path from the DPOF file location:

```
Order folder root:
  DPOF.001           ← DPOF file lives here
  IMAGES/
    photo-001.jpg    ← Image lives here

So: ../IMAGES/photo-001.jpg  ← goes up one level, then into IMAGES/
```

Wait — actually, DPOF.001 is placed directly in the order folder, so the path `../IMAGES/photo-001.jpg` would go **up** from the folder. In practice the generated path is:

```
<IMG SRC = "../IMAGES/photo-001.jpg">
```

This means the DPOF.001 is interpreted as being in a subfolder of the order root, or the controller resolves paths relative to the order folder itself. Verify with your specific controller if image references fail.

### IMG ORG path

The `IMG ORG` field records the original path of the image within the hot folder structure:

```
IMG ORG = "o100456_8x12GLOSS/IMAGES/photo-001.jpg"
```

This is an audit/tracking field — it records where the image came from within the hot folder, using the `o`-prefixed folder name as submitted.

## Complete Annotated Example

Order number `100456`, product `8x12GLOSS`, 2 images, 1 copy each.

```
[HDR]
GEN REV = 01.00                              ← DPOF format version
GEN CRT = "OHD" 1.00                         ← Created by OrderHub Downloader v1.00
GEN DTM = 2026:02:21:15:30:45               ← Timestamp of file generation
USR NAM = "Jane Smith"                       ← Customer name
USR CID = "100456"                           ← Order number (used as customer ID)
AUTO CORRECT = 1                             ← Enable auto colour correction
VUQ RGN = BGN                               ← Begin vendor-specific section
VUQ VNM = "NORITSU KOKI" -ATR "QSS-3901"   ← Target controller vendor + model
VUQ VER = 01.00                             ← Vendor qualifier version
PRT PSL = B -PSIZE "8x12"                  ← Style code B, paper size 8x12 inches
PRT PCH = 145                               ← Use channel 145
VUQ RGN = END                               ← End vendor-specific section

[JOB]
PRT PID = 001                               ← First print job
PRT TYP = STD                               ← Standard print
PRT QTY = 1                                 ← 1 copy
IMG FMT = EXIF2 -J                          ← JPEG with EXIF data
<IMG SRC = "../IMAGES/photo-001.jpg">       ← Image file reference
VUQ RGN = BGN
VUQ VNM = "NORITSU KOKI" -ATR "QSS-3901"
VUQ VER = 01.00
PRT CVP1 = 1 -STR ""                        ← Custom var 1: blank
PRT CVP2 = 1 -STR "100456"                 ← Custom var 2: order reference
IMG ORG = "o100456_8x12GLOSS/IMAGES/photo-001.jpg"  ← Original path
VUQ RGN = END

[JOB]
PRT PID = 002                               ← Second print job
PRT TYP = STD
PRT QTY = 1
IMG FMT = EXIF2 -J
<IMG SRC = "../IMAGES/photo-002.jpg">
VUQ RGN = BGN
VUQ VNM = "NORITSU KOKI" -ATR "QSS-3901"
VUQ VER = 01.00
PRT CVP1 = 1 -STR ""
PRT CVP2 = 1 -STR "100456"
IMG ORG = "o100456_8x12GLOSS/IMAGES/photo-002.jpg"
VUQ RGN = END
```

## Noritsu vs Epson Differences

The DPOF format itself is standard, but the **vendor metadata** fields differ between manufacturers.

| Field | Noritsu | Epson |
|-------|---------|-------|
| `VUQ VNM` | `"NORITSU KOKI"` | e.g., `"EPSON"` |
| `-ATR` | e.g., `"QSS-3901"` | e.g., `"SC-P900"` |
| `PRT PSL` code | `B` (borderless standard) | Controller-specific |
| `PRT PCH` | Noritsu channel numbers | Epson channel numbers |

The controller `type` field (`noritsu` / `epson`) is stored for future use — currently all DPOF generation uses the same template, with differences only in the `vendorName` and `vendorAttribute` values configured per controller.

## Timestamp Format

The `GEN DTM` timestamp uses the format:

```
YYYY:MM:DD:HH:MM:SS
```

Generated by `DPOFGenerator.formatTimestamp()`:

```
2026:02:21:15:30:45
 │    │  │  │  │  └── seconds (00-59)
 │    │  │  │  └───── minutes (00-59)
 │    │  │  └──────── hours (00-23)
 │    │  └─────────── day (01-31)
 │    └────────────── month (01-12)
 └─────────────────── year (4 digits)
```
